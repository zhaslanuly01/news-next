## О проекте

Приложение получает новости из **News API** и предоставляет удобный веб-интерфейс для:

- просмотра списка новостей,
- фильтрации по категории и дате,
- поиска,
- просмотра деталей статьи,
- открытия оригинала статьи в новой вкладке / встроенном viewer,
- добавления в избранное,
- импорта/экспорта статей в JSON,
- скачивания изображения статьи,
- получения браузерных уведомлений (при наличии разрешения).

## Архитектура проекта

Проект организован в стиле **FSD (Feature-Sliced Design)**.

### Почему выбран такой подход

FSD помогает:

- разделить ответственность между слоями,
- упростить поддержку и масштабирование,
- сделать код более предсказуемым и переиспользуемым.

### Слои

#### `app`

Глобальная конфигурация приложения:

- `providers` (Redux Store Provider, Theme Provider)
- `router` / layout (`app` router layouts, pages)
- `bootstrap` (инициализация session/notifications на клиенте)
- глобальные стили

#### `pages` / `app` (в зависимости от структуры Next.js)

Экранные страницы:

- `login`
- `news`
- `news/[id]` (или slug-based route)
- `favorites`

> Если используется **App Router**, роль pages выполняют сегменты в `src/app/*`.

#### `widgets`

Составные UI-блоки страниц:

- панель фильтров
- список новостей
- карточка статьи
- блок действий на деталях статьи
- блок контента статьи

#### `features`

Пользовательские сценарии (use cases):

- demo-login / logout
- переключение избранного
- скачивание изображения
- экспорт JSON
- импорт JSON-файла
- фильтры новостей (UI + типы/константы)

#### `entities`

Бизнес-сущности:

- `article`
- `favorite`
- `session`
- `file` (типы для файловых операций)

#### `shared`

Переиспользуемая инфраструктура:

- `api` (RTK Query baseApi)
- `config` (ENV)
- `lib` (notifications, files, hooks, utils)
- `types`
- `ui` (shadcn wrappers / shared components)

## Обязательные функции и их реализация

Ниже описаны ключевые обязательные функции и как они реализованы в проекте (в веб-версии на Next.js).

### 1) Отправка файлов с устройства (обязательное)

Реализовано через фичу **`features/upload-file`**.

#### Что делает

- позволяет выбрать файл с устройства (`<input type="file" />`)
- проверяет размер (лимит до 10MB)
- проверяет тип файла (`.json` / mime type)
- читает содержимое файла (`FileReader`)
- парсит JSON
- валидирует структуру
- импортирует статью в избранное (`favorites`)
- сохраняет обновлённый список в `localStorage`

#### Обработка ошибок

Поддерживаются и маппятся пользовательские ошибки:

- файл слишком большой
- отмена выбора пользователем / файл не выбран
- неверный тип файла
- некорректный JSON
- неверная структура JSON
- ошибка чтения файла
- неизвестная ошибка

#### Почему это считается “отправкой файла”

В учебных/демо задачах этот сценарий покрывает пользовательский поток выбора и обработки файла с устройства.
При необходимости легко заменить локальную обработку на реальный HTTP upload к серверу (через `fetch` + `FormData` / отдельный endpoint в Next.js API routes).

---

### 2) Загрузка (скачивание) файлов с сервера + обработка ошибок (обязательное)

Реализовано через фичу **`features/article-download-image`** + `shared/lib/files`.

#### Что делает

- берёт `urlToImage` из статьи
- скачивает файл по URL (`fetch`)
- преобразует ответ в `Blob`
- создаёт ссылку через `URL.createObjectURL`
- инициирует скачивание через `<a download>`
- освобождает object URL после завершения

#### Обработка ошибок

- отсутствие изображения у статьи
- ошибка загрузки файла по URL
- ошибка преобразования ответа в файл
- блокировка/ограничения браузера
- отображение ошибок через `toast` / alert / текст под кнопкой

---

### 3) Хранение данных на устройстве

#### Избранное

- `localStorage`
- хранение списка избранных статей (`favorite_articles_v1`)

#### Сессия / настройки сессии

- `localStorage` / cookie (в demo-режиме)
- хранение признака авторизации и клиентских настроек

> Для production-auth рекомендуется использовать `httpOnly cookies` и серверную сессию/JWT.

---

### 4) Авторизация / сессия

Реализовано через:

- `features/demo-login`
- `features/logout`
- `entities/session`
- `app/bootstrap/SessionBootstrap`

#### Что происходит

- при запуске клиентской части восстанавливается demo-сессия из `localStorage`
- пока сессия инициализируется — показывается loader/skeleton
- если не авторизован — страница `Login`
- если авторизован — основной интерфейс (news + favorites)

---

### 5) Работа с API

Реализовано через:

- `shared/api/baseApi`
- `entities/article/api/articleApi`

#### Почему RTK Query

Выбран RTK Query, потому что он:

- уменьшает количество boilerplate-кода
- удобно управляет loading/error/data
- поддерживает кэширование
- хорошо интегрируется с Redux Toolkit

---

### 6) Уведомления

Реализовано через `shared/lib/notifications` + `NotificationsBootstrap`.

#### Возможности

- запрос разрешения на уведомления через Web Notifications API
- локальные уведомления в браузере:

  - при добавлении в избранное
  - при появлении новых новостей

> Поддержка уведомлений зависит от браузера и пользовательских разрешений.

## Установка и настройка

### 1. Клонировать репозиторий

```bash
git clone <URL_репозитория>
cd <название-проекта>
```

### 2. Установить зависимости

```bash
npm install
```

или

```bash
yarn
```

или

```bash
pnpm install
```

### 3. Создать `.env.local`

Создайте файл `.env.local` в корне проекта и добавьте:

```env
NEXT_PUBLIC_NEWS_API_BASE_URL=https://newsapi.org/v2
NEXT_PUBLIC_NEWS_API_KEY=ВАШ_NEWS_API_KEY
```

> Если `NEXT_PUBLIC_NEWS_API_KEY` не задан, приложение покажет предупреждение в консоли / UI.

### 4. Получить API ключ

Получите ключ на News API и подставьте в `.env.local`.

## Особенности реализации (пояснения по решениям)

### 1. Разделение на `entities / features / widgets / app`

Это упрощает поддержку:

- бизнес-модели не смешиваются со страницами
- пользовательские сценарии инкапсулированы в `features`
- страницы собираются из готовых блоков

### 2. Redux Toolkit + RTK Query

Использован единый подход для:

- локального состояния (`session`, `favorites`)
- серверного состояния (`articleApi`)

Это снижает сложность и делает код более единообразным.

### 3. shadcn/ui + Tailwind CSS

Использован подход с composable UI-компонентами:

- быстрый старт и удобная стилизация
- переиспользуемые базовые компоненты (`Button`, `Card`, `Dialog`, `Input`, `Badge`)
- хорошая масштабируемость UI-слоя

### 4. Явная обработка ошибок файлового импорта

В `upload-file` сделан отдельный формат ошибок (`FileTransferError`) и маппинг кодов в понятные сообщения для пользователя.

### 5. Клиентский bootstrap

Отдельные `SessionBootstrap` и `NotificationsBootstrap` позволяют:

- не перегружать layout/page
- централизованно запускать init-логику
- легко добавлять новые bootstrap-модули
